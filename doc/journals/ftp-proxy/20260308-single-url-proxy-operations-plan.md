# 단일 URL 유지 및 프록시 운영 계획

**날짜**: 2026-03-08

## 1. 진행 사항

- 클라우드에 배포된 FastAPI 서버가 애플리케이션 API와 프록시 작업을 함께 처리하는 현재 구조를 운영 관점에서 정리함
- 회사 리소스 제약 사항 확인: CPU 4개 한도 내에서 운영되며, 과사용으로 쿼터가 소진되면 일정 시간 재확보를 기다려야 하는 상황을 전제로 검토함
- 환경별 주소 후보(`dev`, `stg`, `prod`)를 별도로 노출하거나 전환하는 방향은 보류하고, 사용자에게는 단일 URL만 유지하기로 결정함
- 클라이언트가 응답 주체를 의식하지 않아도 되도록 public entrypoint는 하나로 고정하고, 운영상 대응은 동일 URL 뒤쪽에서 처리하는 방향으로 정리함
- base URL 전환 자체보다, 단일 URL 안정성 확보를 위한 부하 제어와 운영 가시성 확보가 핵심 과제라는 점을 명확히 함

## 3. 다음 단계

- 사용자 노출 주소는 단일 URL로 유지하고, `dev/stg/prod` 간 base URL 전환 시나리오는 운영 전략에서 제외
- CPU 쿼터 소진 상황에서도 동일 URL을 유지할 수 있도록 요청 제한(rate limit), 동시 실행 제한, 작업 큐 도입 가능성 검토
- `/health` 외에 프록시 부하 상태를 반영하는 readiness 지표(예: 현재 작업 수, 큐 길이, CPU 사용률 임계치) 정의
- 운영자는 실제 처리 노드 또는 인스턴스를 식별할 수 있어야 하므로 내부 로그 또는 응답 헤더(예: `X-Served-By`) 설계
- 프록시 작업이 CPU bound인지 I/O bound인지 먼저 계측하고, 애플리케이션 API와 프록시 작업을 동일 프로세스에서 계속 처리할지 분리할지 판단
- 필요한 경우 단일 URL 뒤에 동일 스펙의 운영 인스턴스를 추가하는 scale-out 방안 검토
- 우선순위 제안:
  1. 단일 URL 기준의 부하 제어 정책 정의
  2. 프록시 작업 특성 계측
  3. readiness/observability 지표 추가
  4. 필요 시 동일 URL 뒤 인스턴스 확장 검토
